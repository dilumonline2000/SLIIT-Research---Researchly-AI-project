## Model details

### `citation_ner/`
- **Architecture**: spaCy `en_core_web_sm` + transformer NER head
- **Training data**: 3,866 citation strings from 539 SLIIT papers
- **Targets**: `AUTHOR`, `YEAR`, `TITLE`, `VENUE`, `DOI`
- **Metric**: F1 = **99.05%** (target: ≥0.85)
- **Loaded by**: `app/models/citation_ner.py`

### `sbert_plagiarism/`
- **Architecture**: Sentence-Transformer (all-MiniLM-L6-v2 base, fine-tuned)
- **Training data**: 500 SLIIT abstract pairs with triplet loss
- **Output**: 384-dim L2-normalized embeddings
- **Metric**: Accuracy = **100%** at threshold 0.7 (target: ≥0.75)
- **Loaded by**: `app/routers/plagiarism.py`, and reused as the encoder for
  the gap & proposal retrievers below

### `trained_gap_analyzer/`
- **What it is**: a pickled SBERT retrieval index, not a fresh model
- **Files**:
  - `gap_index.pkl` — `{embeddings: float32[553, 384], records, version}`
  - `metadata.json` — stats: `n_gaps`, `n_unique_papers`, `gap_type_counts`
- **Loaded by**: `app/services/gap_analyzer.py`
- **Regenerate**: `python ../training/train_gap_analyzer.py`

### `trained_proposal_retriever/`
- **What it is**: SBERT retrieval index over 3,858 SLIIT proposal exemplars
- **Files**:
  - `proposal_index.pkl` — `{embeddings: float32[3858, 384], records, version}`
  - `metadata.json` — stats
- **Loaded by**: `app/services/proposal_retriever.py`
- **Regenerate**: `python ../training/train_proposal_retriever.py`

## Why the gap & proposal indices are not "trained from scratch"

We re-use the SLIIT-fine-tuned SBERT (`sbert_plagiarism/`) as a feature
extractor. The "training" step is encoding the corpus into a dense matrix,
which is the standard recipe for retrieval-augmented systems.

This gives us:
- **Grounded answers** — every result cites a real SLIIT paper
- **Fast inference** — sub-100 ms per query (one matmul against ~553×384
  or ~3858×384)
- **Low maintenance** — adding a new paper is a single re-encode, no
  hyper-parameter tuning, no overfitting risk
